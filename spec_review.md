# Skills Manager 规格审查规范

本文档定义 Skills Manager 的产品规格、行为合同和实施计划如何接受语义审查。它不替代 GitHub Epic、FR、Sub-issues 与原生依赖关系；这些宿主对象仍是产品规划真相。

## 目的与边界

规格审查回答的是：需求是否足够明确、安全、可实现且可验证，可以进入实现阶段。

它不判断实现代码是否正确，也不直接授权合并或发布。实现完成后仍须按 [code_review.md](code_review.md) 进行代码审查，并通过当前 PR head 的 CI、host attestation 与 merge-ready 检查。

## 何时需要审查

以下事项进入实现前必须完成规格审查：

- 改变用户可观察行为、数据模型、持久化格式或迁移路径。
- 涉及 SSOT、SQLite、操作日志、文件导入、删除、备份或恢复。
- 涉及 Symlink、Copy、Fork、Agent 分发范围或冲突处理。
- 涉及 Provider、远程来源身份、更新判断或网络失败语义。
- 涉及权限、安全边界、并发、一致性、签名、公证、Sparkle 或发布。
- 跨越多个模块、Work Item 或外部系统的合同变更。

纯措辞、链接、注释、格式化或不改变行为的机械维护可以标记为 `not-applicable`，但必须记录理由和替代验证方式。

## 最小输入

审查前至少提供：

- 唯一 GitHub FR 或 Work Item，以及 parent、Sub-issues 和依赖关系。
- 用户价值、成功体验、失败或不可用状态、敏感数据边界和非目标。
- 明确的行为合同、状态变化、错误语义、兼容与迁移策略。
- 可执行的验收标准和验证策略。
- 相关权威 locator；引用现有事实来源，不复制第二份定义。
- 若规格通过 PR 修改：PR、branch 与当前 head SHA。

输入不完整时先返回 `fallback`，说明应回到 Story shaping、FR 拆分或 Work Item 细化中的哪一层，不以猜测补齐产品决策。

## 审查维度

| 维度 | 通过信号 | 阻断信号 |
| --- | --- | --- |
| 用户价值与范围 | 成功、失败、不可用状态和非目标明确，未越出上位 FR | 只有功能清单，或混入相邻 Work Item |
| 身份与事实来源 | UUID、slug、来源身份、SSOT 与数据库权威边界唯一 | 路径、名称、Provider 别名被混作稳定身份 |
| 数据与一致性 | 事务边界、幂等、并发、操作日志及中断恢复可判定 | SQLite 与文件系统可能形成无恢复路径的部分状态 |
| 文件安全 | 路径规范化、根目录证明、链接策略、资源上限和原子落盘明确 | 允许路径穿越、越界链接、选择性覆盖或半成品 |
| 分发语义 | 全局与 Agent 专属转换、重复避免、启用/停用和删除本体边界明确 | 同一 Skill 可在全局和专属目录以不同版本共存 |
| Copy 与 Fork | drift 检测、显式 Fork、身份分离和冲突选择可验证 | 静默覆盖用户修改或隐式改变全局语义 |
| Provider 与更新 | 真实来源去重、revision、哈希、离线与限流行为明确 | Provider 列表或显示名称成为来源真相 |
| UI 与可访问性 | 进行中、成功、空、冲突、失败、恢复状态均有反馈 | 后台操作完成前显示成功，或破坏性操作无确认/恢复入口 |
| 兼容与迁移 | 旧状态保留、失败可重试、回滚与版本边界明确 | 迁移失败后产生双重权威或不可逆数据损失 |
| 发布与安全 | secrets、签名、公证、Sparkle、升级和发布后验证责任明确 | 密钥进入仓库，或发布流程无法证明产物来源与完整性 |
| 验证策略 | 自动化测试、fixture、集成检查和必要人工验收覆盖主要风险 | 只写“补测试”“手动验证”而没有可执行证据 |

## Finding 与结论

Finding 使用以下优先级：

- `P0`：可能造成越权、数据破坏、密钥泄漏或不可恢复发布事故。
- `P1`：核心行为、身份、一致性或迁移合同错误，必须在实现前修正。
- `P2`：会造成明显边界遗漏、回归风险或验证缺口。
- `P3`：非阻断的清晰度、可维护性或后续改进建议。

正式结论只能是：

- `allow`：没有未解决的 P0–P2，规格已达到 implementation-ready。
- `block`：存在明确的 P0–P2，必须修改当前规格。
- `fallback`：输入或上游产品决策不足，无法形成可靠结论。

结论至少记录审查类型、GitHub locator、审查对象版本或 PR head、findings、验证要求、剩余风险与下一入口。`allow` 只表示可以实现，不表示实现正确或可以合并。

## 新鲜度与 Loom

- 规格、验收、风险边界或依赖关系改变后，旧结论立即过期。
- PR 形式的规格审查必须绑定当前 head；head 改变后重新确认。
- `loom pre-review` 只验证事实链完整，不产生语义结论。
- 正式结论发布到 GitHub 认可的 review/check 入口，并由 Loom host attestation 消费。
- 不创建 `.loom/reviews/**`、repo-local spec record、current、progress、shadow 或 closeout carrier 作为替代真相。
